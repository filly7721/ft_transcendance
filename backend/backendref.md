# Backend Reference — NestJS + Prisma

A reference for how to structure and grow the `backend/` of ft_transcendance.
The stack: **NestJS 11 (Express)** + **Prisma** + **TypeScript**.

---

## 1. Core principles

1. **Feature modules, not layer folders.** Group code by domain (`users/`, `auth/`, `games/`), not by kind (`controllers/`, `services/`). Everything about "users" lives in one folder — easy to find, easy to delete.
2. **Thin controllers, fat services.** Controllers only parse/validate input and shape the HTTP response. All business logic lives in services. A service should be callable from a controller, a websocket gateway, or a test without change.
3. **Prisma is the data layer.** One shared `PrismaService`. Services call Prisma directly (no extra repository layer needed at this project's size — Prisma's client *is* the repository).
4. **Validate at the edge.** Every request body goes through a DTO with `class-validator` decorators and a global `ValidationPipe`. Nothing untyped enters a service.
5. **Never leak DB entities raw.** What you return over HTTP is a deliberate choice (a response DTO or a Prisma `select`). Never `return user` when `user` contains `passwordHash`.
6. **Config is validated once at boot.** Missing `DATABASE_URL` or `JWT_SECRET` should crash the app at startup, not at first request.
7. **One source of truth for the schema.** `prisma/schema.prisma` defines the models; migrations are committed to git; the generated client gives you types everywhere.

## 2. Suggested file structure

```
backend/
├── prisma/
│   ├── schema.prisma           # models, datasource, generator
│   ├── migrations/             # generated by `prisma migrate dev` — commit these

├── src/
│   ├── main.ts                 # bootstrap: pipes, CORS, prefix, port

│   │
│   ├── config/
│   │   └── env.validation.ts   # validates process.env at startup
│   │
│   ├── prisma/
│   │   ├── prisma.module.ts    # @Global() module exporting PrismaService
│   │   └── prisma.service.ts   # extends PrismaClient, connects on init
│   │
│   ├── common/                 # cross-cutting, domain-agnostic helpers
│   │   ├── decorators/         # e.g. @CurrentUser()
│   │   ├── guards/             # e.g. JwtAuthGuard (if not in auth/)
│   │   ├── filters/            # e.g. PrismaExceptionFilter
│   │   └── interceptors/
│   │
│   ├── users/
│   │   ├── users.module.ts
│   │   ├── users.controller.ts
│   │   ├── users.service.ts
│   │   └── dto/
│   │       ├── create-user.dto.ts
│   │       ├── update-user.dto.ts
│   │       └── user-response.dto.ts
│   │
│   ├── auth/
│   │   ├── auth.module.ts
│   │   ├── auth.controller.ts   # /auth/login, /auth/callback (42 OAuth), /auth/refresh
│   │   ├── auth.service.ts
│   │   ├── strategies/          # jwt.strategy.ts, ft.strategy.ts
│   │   └── dto/
│   │
│   └── games/                   # match history, live games
│       ├── games.module.ts
│       ├── games.controller.ts  # REST: history, leaderboard
│       ├── games.service.ts     # rules-agnostic persistence + queries
│       ├── games.gateway.ts     # WebSocket: live game events
│       └── dto/
├── test/                        # e2e tests (supertest)
├── .env                         # local secrets — never commit
├── .env.example                 # committed template of required vars
└── package.json
```

Growth rule: **a new feature = a new folder under `src/` with its own `*.module.ts`**, registered in `app.module.ts`. Modules should depend on other modules' *services* (via imports/exports), never reach into their files directly.

What `common/` is for: things used by 3+ modules and tied to no domain. If it's only used by one feature, keep it inside that feature's folder.

## 3. Prisma setup

Install:

```bash
npm i @prisma/client
npm i -D prisma
npx prisma init          # creates prisma/schema.prisma and .env DATABASE_URL
```

Daily workflow:

```bash
npx prisma migrate dev --name add_games   # edit schema → create+apply migration → regen client
npx prisma studio                          # browse the DB in a UI
npx prisma migrate deploy                  # apply committed migrations (prod/CI)
```

### `prisma/schema.prisma` (example)

```prisma
generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"      // or "sqlite" for local dev
  url      = env("DATABASE_URL")
}

model User {
  id           Int      @id @default(autoincrement())
  email        String   @unique
  username     String   @unique
  passwordHash String?                    // null when using 42 OAuth only
  avatarUrl    String?
  createdAt    DateTime @default(now())
  updatedAt    DateTime @updatedAt

  gamesAsP1 Game[] @relation("player1")
  gamesAsP2 Game[] @relation("player2")
}

enum GameStatus {
  IN_PROGRESS
  FINISHED
  ABANDONED
}

model Game {
  id        Int        @id @default(autoincrement())
  status    GameStatus @default(IN_PROGRESS)
  score1    Int        @default(0)
  score2    Int        @default(0)
  winnerId  Int?
  createdAt DateTime   @default(now())
  endedAt   DateTime?

  player1   User @relation("player1", fields: [player1Id], references: [id])
  player1Id Int
  player2   User @relation("player2", fields: [player2Id], references: [id])
  player2Id Int

  @@index([player1Id])
  @@index([player2Id])
}
```

Conventions: singular PascalCase model names, camelCase fields, explicit `@@index` on foreign keys you query by, enums for closed sets of states.

### `src/prisma/prisma.service.ts`

```ts
import { Injectable, OnModuleInit, OnModuleDestroy } from '@nestjs/common';
import { PrismaClient } from '@prisma/client';

@Injectable()
export class PrismaService
  extends PrismaClient
  implements OnModuleInit, OnModuleDestroy
{
  async onModuleInit() {
    await this.$connect();
  }

  async onModuleDestroy() {
    await this.$disconnect();
  }
}
```

### `src/prisma/prisma.module.ts`

`@Global()` so every feature module can inject `PrismaService` without re-importing:

```ts
import { Global, Module } from '@nestjs/common';

@Global()
@Module({
  providers: [PrismaService],
  exports: [PrismaService],
})
export class PrismaModule {}
```

## 4. Example feature module: `users/`

### `dto/create-user.dto.ts`

```ts
import { IsEmail, IsString, Length, Matches } from 'class-validator';

export class CreateUserDto {
  @IsEmail()
  email: string;

  @IsString()
  @Length(3, 20)
  @Matches(/^[a-zA-Z0-9_-]+$/, {
    message: 'username may only contain letters, digits, _ and -',
  })
  username: string;

  @IsString()
  @Length(8, 72)
  password: string;
}
```

(`npm i class-validator class-transformer` — enforced by the global `ValidationPipe` in `main.ts`.)

### `dto/update-user.dto.ts`

```ts
import { PartialType, OmitType } from '@nestjs/mapped-types';
import { CreateUserDto } from './create-user.dto';

export class UpdateUserDto extends PartialType(
  OmitType(CreateUserDto, ['password'] as const),
) {}
```

### `users.service.ts`

```ts
import {
  ConflictException,
  Injectable,
  NotFoundException,
} from '@nestjs/common';
import { Prisma } from '@prisma/client';
import * as bcrypt from 'bcrypt';
import { PrismaService } from '../prisma/prisma.service';
import { CreateUserDto } from './dto/create-user.dto';

// Central definition of the "safe to expose" shape — passwordHash never selected.
export const publicUserSelect = {
  id: true,
  email: true,
  username: true,
  avatarUrl: true,
  createdAt: true,
} satisfies Prisma.UserSelect;

@Injectable()
export class UsersService {
  constructor(private readonly prisma: PrismaService) {}

  async create(dto: CreateUserDto) {
    const passwordHash = await bcrypt.hash(dto.password, 10);
    try {
      return await this.prisma.user.create({
        data: { email: dto.email, username: dto.username, passwordHash },
        select: publicUserSelect,
      });
    } catch (e) {
      // P2002 = unique constraint violation
      if (
        e instanceof Prisma.PrismaClientKnownRequestError &&
        e.code === 'P2002'
      ) {
        throw new ConflictException('email or username already taken');
      }
      throw e;
    }
  }

  async findOne(id: number) {
    const user = await this.prisma.user.findUnique({
      where: { id },
      select: publicUserSelect,
    });
    if (!user) throw new NotFoundException(`user ${id} not found`);
    return user;
  }

  // Used internally by auth — the ONLY place the hash comes out of the DB.
  findByEmailWithHash(email: string) {
    return this.prisma.user.findUnique({ where: { email } });
  }
}
```

### `users.controller.ts`

```ts
import {
  Body,
  Controller,
  Get,
  Param,
  ParseIntPipe,
  Post,
} from '@nestjs/common';
import { UsersService } from './users.service';
import { CreateUserDto } from './dto/create-user.dto';

@Controller('users')
export class UsersController {
  constructor(private readonly usersService: UsersService) {}

  @Post()
  create(@Body() dto: CreateUserDto) {
    return this.usersService.create(dto);
  }

  @Get(':id')
  findOne(@Param('id', ParseIntPipe) id: number) {
    return this.usersService.findOne(id);
  }
}
```

Note what the controller does *not* do: no try/catch, no Prisma, no hashing. `NotFoundException` etc. become proper HTTP status codes automatically.

### `users.module.ts`


## 5. Bootstrap and root module

### `src/main.ts`

```ts
import { ValidationPipe } from '@nestjs/common';
import { NestFactory } from '@nestjs/core';
import { AppModule } from './app.module';

async function bootstrap() {
  const app = await NestFactory.create(AppModule);

  app.setGlobalPrefix('api');
  app.enableCors({
    origin: process.env.FRONTEND_URL ?? 'http://localhost:3000',
    credentials: true,
  });
  app.useGlobalPipes(
    new ValidationPipe({
      whitelist: true,            // strip properties not in the DTO
      forbidNonWhitelisted: true, // 400 on unknown properties
      transform: true,            // plain JSON → DTO class instances
    }),
  );

  await app.listen(process.env.PORT ?? 3001);
}
bootstrap();
```

(Backend on 3001 since Next.js dev takes 3000 — adjust to your setup.)

### `src/config/env.validation.ts`

```ts
import { plainToInstance } from 'class-transformer';
import { IsInt, IsString, IsUrl, validateSync } from 'class-validator';

class EnvVars {
  @IsString()
  DATABASE_URL: string;

  @IsString()
  JWT_SECRET: string;

  @IsInt()
  PORT: number = 3001;
}

export function validateEnv(config: Record<string, unknown>) {
  const validated = plainToInstance(EnvVars, config, {
    enableImplicitConversion: true,
  });
  const errors = validateSync(validated, { skipMissingProperties: false });
  if (errors.length > 0) throw new Error(errors.toString());
  return validated;
}
```

Keep `.env.example` in sync with `EnvVars` so anyone cloning knows what to provide.

## 6. WebSockets (live games)

Gateways are the websocket equivalent of controllers — same rule applies: **the gateway translates socket events to service calls, game logic lives in a plain service/engine class**. Your frontend already has the super-tic-tac-toe rules engine designed with "backend seams"; the server-side twin of it should be a pure TS class with no Nest/socket imports, so the gateway just wires messages to it.

```ts
// games/games.gateway.ts (sketch)
import {
  SubscribeMessage,
  WebSocketGateway,
  MessageBody,
  ConnectedSocket,
} from '@nestjs/websockets';
import { Socket } from 'socket.io';
import { GamesService } from './games.service';

@WebSocketGateway({ cors: { origin: process.env.FRONTEND_URL } })
export class GamesGateway {
  constructor(private readonly gamesService: GamesService) {}

  @SubscribeMessage('game:move')
  async handleMove(
    @ConnectedSocket() client: Socket,
    @MessageBody() payload: { gameId: number; move: unknown },
  ) {
    const state = await this.gamesService.applyMove(
      payload.gameId,
      client.data.userId,
      payload.move,
    );
    client.to(`game:${payload.gameId}`).emit('game:state', state);
    return state; // ack to the sender
  }
}
```

Requires `npm i @nestjs/websockets @nestjs/platform-socket.io socket.io`.
Key practices: authenticate the socket on connection (verify JWT in the handshake, stash `userId` in `client.data`), use rooms per game, **never trust client-sent state** — clients send *moves*, the server owns the authoritative game state and validates every move against it.

## 7. Error handling

- Throw Nest's `HttpException` subclasses (`NotFoundException`, `ConflictException`, `ForbiddenException`…) from services; Nest maps them to responses.
- Translate known Prisma error codes where they occur (`P2002` unique violation → 409, `P2025` record not found → 404). If that gets repetitive, promote it to a global exception filter in `common/filters/prisma-exception.filter.ts`.
- Anything unhandled becomes a 500 with no internal details leaked — that's the correct default; don't catch-and-swallow.

## 8. Testing shape

- **Unit tests** (`*.spec.ts` next to the file): test services with a mocked `PrismaService`. Game rules engines are pure classes — test them with zero mocks.
- **e2e tests** (`test/*.e2e-spec.ts`): boot the app against a throwaway SQLite/Postgres DB, hit real HTTP with supertest. A handful of these covering auth + one CRUD flow catches more than a pile of mocks.

## 9. Checklist for adding a new feature

1. Add/extend models in `schema.prisma` → `npx prisma migrate dev --name <what_changed>`.
2. Create `src/<feature>/` with module, service, controller (and gateway if realtime).
3. Define DTOs for every input; pick an explicit `select` for every output.
4. Register the module in `app.module.ts`.
5. New env vars? Add to `EnvVars` and `.env.example`.
6. Errors you expect (duplicates, missing rows) → typed `HttpException`s in the service.
